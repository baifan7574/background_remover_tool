# 🛠️ 系统检查与修复标准操作流程

## 📋 每次修改前必须执行的检查清单

### 🔍 第一步：全面系统检查（5分钟）

```powershell
# 1. 服务器状态检查
Invoke-RestMethod -Uri "http://localhost:5000/api/auth/login" -Method POST -Body (@{email="test1234@test.com"; password="123456"} | ConvertTo-Json) -ContentType "application/json"

# 2. 数据库连接检查
$email = "test$((Get-Date).ToString('HHmmss'))@test.com"
Invoke-RestMethod -Uri "http://localhost:5000/api/auth/register" -Method POST -Body (@{email=$email; password="123456"; name="测试"} | ConvertTo-Json) -ContentType "application/json"

# 3. 核心API检查
$token = (登录获取的token)
Invoke-RestMethod -Uri "http://localhost:5000/api/auth/profile" -Method GET -Headers @{Authorization="Bearer $token"} -ContentType "application/json"
Invoke-RestMethod -Uri "http://localhost:5000/api/tools/usage-stats" -Method GET -Headers @{Authorization="Bearer $token"} -ContentType "application/json"
```

### 📊 第二步：记录现状基线

创建检查记录文件：
```markdown
## 系统现状检查记录
- 检查时间：[当前时间]
- 服务器状态：[正常/异常]
- 用户登录：[正常/异常]
- 数据库：[正常/异常]
- API端点：[正常/异常]
- 前端界面：[正常/异常]
```

## 🎯 问题分析与修复流程

### 📝 第三步：明确问题定义

在修改代码前，必须清楚回答：
1. **具体问题是什么？**（用户反馈的详细描述）
2. **影响范围多大？**（哪些用户/功能受影响）
3. **问题严重程度？**（致命/严重/一般/轻微）
4. **是否必须立即修复？**（能否等到下次更新）

### 🔧 第四步：最小化修复方案

制定修复方案时必须考虑：
- ✅ **只修改有问题的代码**
- ✅ **不影响正常工作的功能**
- ✅ **修改范围最小化**
- ✅ **风险可控**

### ⚠️ 禁止行为清单

**绝对不要做：**
- ❌ 重构正常工作的代码
- ❌ 修改多个文件同时解决多个问题
- ❌ 为了代码美观而修改正常功能
- ❌ 没有明确问题就动手改代码

## 🧪 修改后验证流程

### ✅ 第五步：全面功能验证

修改完成后必须验证：

#### 1. 核心功能验证
```powershell
# 用户注册登录流程
# 用户资料获取
# 使用统计显示
# 权限检查功能
# 前端界面显示
```

#### 2. 数据完整性验证
```powershell
# 新用户数据保存
# 老用户数据读取
# 使用次数统计
# 云端数据库同步
```

#### 3. 用户体验验证
```powershell
# 页面加载速度
# 按钮响应状态
# 错误提示信息
# 整体使用流程
```

### 📋 第六步：问题解决确认

确认以下问题都得到解决：
- [ ] 原始问题已修复
- [ ] 没有产生新问题
- [ ] 所有功能正常
- [ ] 用户体验良好
- [ ] 系统稳定运行

## 🚨 应急处理流程

### 如果修改后出现问题：

1. **立即停止服务**
2. **回滚到修改前状态**
3. **分析问题原因**
4. **制定新的修复方案**
5. **重新测试验证**

### 如果用户不满意：

1. **认真听取意见**
2. **诚恳道歉**
3. **分析问题根源**
4. **承诺改进**
5. **实际行动证明**

## 🎯 核心原则总结

**记住：**
- 稳定第一，功能第二
- 用户第一，技术第二
- 长期第一，短期第二
- 质量第一，速度第二

**永远不要：**
- 让项目陷入修复循环
- 牺牲稳定性追求完美
- 忽视用户感受
- 重复同样的错误
